=================== STEP 0 — UNDERSTAND THE TASK BEFORE WRITING ANY CODE ===================
Do NOT jump to a design. First, genuinely understand what is being evaluated and what a good memory must do. Read, in this order, and take notes in DESIGN_NOTES.md as you go:
  1. CONTRACT.md — what a valid memory package is, what entrypoints/tools you must expose, the rules (no hardcoded answers, no scene-id branches, every prediction returns evidence).
  2. metrics.md — the EXACT metric each track is scored on. Internalize these; your whole design should serve them:
       - Track 1 (object location): given a target object label, return ranked 3D object predictions. PRIMARY = success@1 (your TOP-1 prediction must be a real instance of that label, within ~0.5–2 m of ground truth). Label match is "free" (target_label is given) — the skill is genuine 3D localization + ranking the RIGHT instance first. Also reported: proximity (did you get near at all), recall, mrr.
       - Track 2 (referring): given a natural-language referring expression ("the chair next to the window"), return the 3D location of the referred object. PRIMARY = acc@0.5m (top-1 center within 0.5 m), distance-only (no label needed). The skill is disambiguating WHICH instance among many.
       - Track 3 (OpenEQA): given an open-ended question about the scene, return a short natural-language answer. PRIMARY = llm_match (a Sonnet judge scores your answer vs GT in [0,1]). The judge rewards short, correct, direct answers (OpenEQA style) — not verbose descriptions.
  3. shared_modules.md — the perception/LLM stack you MUST use (shared so all methods are comparable): YOLO-World-L open-vocab detector + ScanNet200 class list, local qwen via ollama (qwen3.5:4b vision describer, qwen3-embedding for embeddings). Build determinism matters (temp0/seed) for the keep/revert loop.
  4. dev_scenes/README.md and the dev_tests/ for each scene/track — read the actual query formats and the expected answer JSON shape. Look at examples/. NEVER edit dev_tests.
  5. The repo evaluator code (repo_root in sandbox_config.json) — read how Track 1/2/3 are actually scored, end to end, so you optimize the real metric and not a guess.
Only once you can state, in DESIGN_NOTES.md, "here is exactly what each track needs and how it is scored" should you start designing.

=================== THE LOOP — how you make progress ===================
This is a git-driven keep/revert auto-research loop. After each change to your design, run:
    python autoresearch_round.py --build-cmd "python starter/build_memory.py" --message "round N: <what you changed and why>"
It builds your memory for all 6 dev scenes, scores them on the FIXED dev tests with the REAL Track 1/2/3 evaluators (+ the Sonnet judge for T3), and then:
  - if your loop_objective IMPROVED over the best so far -> it commits (your change is kept),
  - else -> it reverts to the last good commit (your change is rolled back automatically).
So the git log is your experiment journal: every commit is a genuine improvement. Read the FULL feedback each round — memories/design/_dev_eval/dev_score.json has per_track metrics + proximity + per-query detail. Diagnose before you change: is the loss coverage (object missing) or ranking (object present but mis-ranked)? which track/scene is weakest? Form a hypothesis, then act.

YOU decide the design direction and when to iterate vs. rethink. There is no required number of designs or phases — pursue whatever you believe will score best and generalize. Be a good researcher: try ambitious ideas, but verify everything against the real evals, and keep what genuinely works.

=================== METRICS + OBJECTIVE (what the loop maximizes) ===================
loop_objective = success@1[T1] + acc@0.5m[T2] + llm_match[T3]  −  cost_penalty
  - cost_penalty (SCORED): memory soft-hinge above 50 MB/scene; AND a real-time penalty on BUILD time-per-frame — 0 within 0.2 s/frame (~5 fps), rising and saturating up to 0.5 above it. A deployable memory must ingest the camera stream in real time.
TWO SEPARATE COSTS — both are reported each round; watch BOTH:
  - build_cost.mean_time_per_frame_seconds = how fast you ingest frames (SCORED via cost_penalty). Keep it near-real-time by AMORTIZING heavy models (describe/embed each object ONCE when its track stabilizes), GATING them (run the describer on detection events / keyframes, not every frame), or running them ASYNC — the way real streaming systems (ClawS, DAAAM/Hydra) stay real-time.
  - build_cost.query_latency_ms_mean = the FULL query->answer round-trip (retrieve + reason + answer). REPORTED ONLY, NOT in the objective yet. It exists to catch a specific cheat: making the build look fast by DEFERRING the heavy VLM/embedding work to query time. If build TPF is low but query_latency is high, you have NOT built a real-time memory — you've just hidden the cost where the objective can't see it, and a robot answering many queries would pay it every time. DO NOT dump compute into the query path. The memory should CONTAIN the semantic content when a query arrives; query-time tools should be LIGHT (retrieve / rank / ground / read what is stored). Treat rising query_latency as a red flag even though it doesn't change your score.
Build the semantic content INTO the memory at ingest time, in real time. That is the deployable target.

=================== HOW THE SELF-EVAL RUNS YOUR MEMORY ===================
Per scene, ONE persistent answer-agent is fed that scene's queries one at a time; it answers by calling the native tools you declare in capabilities.json (agent_tools[], each fn(package_dir, args)->dict). Final JSON: T1/T2 -> {"final":{"predictions":[{object_id,label,position_3d,score,evidence}]}}; T3 -> {"final":{"answer":"...","evidence":[...]}}. Whatever your memory's internal form, expose tools that let the agent retrieve / ground / answer. Build a VALID package per scene (method.family="agent_designed"; validate with `python -m spatial_memory_evaluation.memory_package_validator <pkg>`). build_memory.py MUST accept --layout-dir --scene-id --out.
